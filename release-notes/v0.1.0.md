# AI-Infra-Auto-Driven-SKILLS v0.1.0

最近这个仓库刚好到 400 stars，所以把 v0.1.0 tag 打了。之前一直是想到什么就补什么，这次算是把已经比较能用的一批 AI Infra SKILLS 收一下，也顺手写个说明。

先感谢 @haosdent 和 @changhuaixin 两位 contributor。@haosdent 补了 `vllm-sota-humanize-loop`，这样同一套 SOTA loop 不只可以追 SGLang，也可以反过来追 vLLM。@changhuaixin 这波补了三件很硬的东西：`llm-pipeline-analysis`、`llm-serving-capacity-planner`、`model-compute-simulation`。没有这几块，Agent 很容易只停在 profiler 三张表，知道慢，但继续往哪一层、哪一类 kernel、哪块 memory budget 下钻就不清楚了。

![AI-Infra-Auto-Driven-SKILLS repo](https://files.mdnice.com/user/59/ac8d131e-1867-49b7-83f8-28e55dd75908.png)

做这个仓库的原因也比较朴素。最近用 Codex 蹬 SGLang/vLLM 相关任务时，最耗人的地方在流程控制：先 benchmark，别拿不公平的结果乱比；启动日志先看清楚，别一上来就改源码；profile 要把 prefill/decode 分开；改完之后必须回到同一个 workload 复测。以前这些靠人一直盯着，现在我把它们尽量写成 `SKILL.md`，让 Codex / Claude Code 可以自己按步骤走。

项目地址：https://github.com/BBuf/AI-Infra-Auto-Driven-SKILLS

v0.1.0 Release 在这里，想锁定版本的话直接用这个：https://github.com/BBuf/AI-Infra-Auto-Driven-SKILLS/releases/tag/v0.1.0

这套 SKILLS 不挑 runtime，Codex 可以用，Claude Code 也可以用。目录结构就是普通的 `SKILL.md`，把对应目录放到各自的 skill 目录下就行。以 v0.1.0 为例：

```bash
git clone --branch v0.1.0 https://github.com/BBuf/AI-Infra-Auto-Driven-SKILLS.git
cd AI-Infra-Auto-Driven-SKILLS

# Codex
mkdir -p "${CODEX_HOME:-$HOME/.codex}/skills"
ln -s "$PWD/skills/llm-serving-auto-benchmark" "${CODEX_HOME:-$HOME/.codex}/skills/llm-serving-auto-benchmark"
ln -s "$PWD/skills/llm-torch-profiler-analysis" "${CODEX_HOME:-$HOME/.codex}/skills/llm-torch-profiler-analysis"
ln -s "$PWD/skills/sglang-sota-humanize-loop" "${CODEX_HOME:-$HOME/.codex}/skills/sglang-sota-humanize-loop"
ln -s "$PWD/skills/vllm-sota-humanize-loop" "${CODEX_HOME:-$HOME/.codex}/skills/vllm-sota-humanize-loop"
ln -s "$PWD/model-pr-optimization-history" "${CODEX_HOME:-$HOME/.codex}/skills/model-pr-history-knowledge"

# Claude Code
mkdir -p "$HOME/.claude/skills"
ln -s "$PWD/skills/llm-serving-auto-benchmark" "$HOME/.claude/skills/llm-serving-auto-benchmark"
ln -s "$PWD/skills/llm-torch-profiler-analysis" "$HOME/.claude/skills/llm-torch-profiler-analysis"
ln -s "$PWD/skills/sglang-sota-humanize-loop" "$HOME/.claude/skills/sglang-sota-humanize-loop"
ln -s "$PWD/skills/vllm-sota-humanize-loop" "$HOME/.claude/skills/vllm-sota-humanize-loop"
ln -s "$PWD/model-pr-optimization-history" "$HOME/.claude/skills/model-pr-history-knowledge"
```

如果只是试试，不需要一口气全装。比如只想跑服务压测，就先装 `llm-serving-auto-benchmark`；想看 trace，就装 `llm-torch-profiler-analysis` 和 `llm-pipeline-analysis`。要跑完整 SOTA loop，再把 benchmark、profiler、pipeline analysis、model PR history、Humanize/RLCR 相关 skill 都配上。

# 0x1. Core Skills

这版核心 skill 先收 10 个，基本都是我自己在推理框架开发、debug、profile、benchmark 里会反复碰到的场景：

| Skill | 解决的问题 |
| --- | --- |
| `llm-serving-auto-benchmark` | 对 SGLang、vLLM、TensorRT-LLM 或其它 OpenAI-compatible server 做公平的 serving benchmark 搜索。 |
| `llm-serving-capacity-planner` | 从 SGLang/vLLM 启动日志里看 GPU memory、KV cache、request capacity 和 OOM pressure。 |
| `llm-torch-profiler-analysis` | 读 torch profiler trace，输出 kernel、overlap、fuse opportunity 三张表，并把 prefill/decode 分开。 |
| `llm-pipeline-analysis` | 继续往 forward、layer、kernel timeline 下钻，找代表层、anchor kernel 和 Perfetto 时间范围。 |
| `model-compute-simulation` | 根据模型结构估算 operator shapes、FLOPs、MFU，再把 kernel 和 op 对起来看。 |
| `sglang-humanize-review` | 用 2024-2025 SGLang human review 语料做代码审查，让 review 更接近 maintainer 平时会看的点。 |
| `sglang-sota-humanize-loop` | 给一个模型和硬件预算，让 SGLang 在固定 workload/SLA 下追平或超过可复现的最强 competitor。 |
| `vllm-sota-humanize-loop` | 同样的模型级 SOTA loop，不过目标框架换成 vLLM。 |
| `sglang-prod-incident-triage` | 线上 serving 出现 queue growth、timeout、wrong output、crash、hang 时，先抽 replay，再决定下一步 debug。 |
| `model-architecture-diagram` | 找 DeepSeek、GLM、Qwen、Kimi、MiniMax、Step、Hunyuan、Qwen3-VL 等模型的公开原始架构图。 |

旁边还有一个 `model-pr-optimization-history`。我更把它当成模型优化 PR 的本地知识库。SOTA loop 在动源码前会先查它：这个模型家族之前谁动过，动了哪些文件，踩过哪些坑，哪些 PR 关了但思路还能借。这个东西我个人觉得很有用，因为 Agent 很容易重新发现一遍人类已经试过的失败方向。

# 0x2. 两个 SOTA Loop

这次最想说的是两个 SOTA loop。

`sglang-sota-humanize-loop` 做的事情比较直接：给一个模型和硬件预算，先跑一轮固定公平 benchmark，再决定 SGLang 有没有必要继续追。如果有差距，就 profile、看 pipeline、改源码、复测。benchmark 表不能只在最后贴 PR 时露个脸，它得回答下一轮该不该 patch、该 patch 哪里、patch 后到底有没有变快。

![SGLang SOTA Humanize Loop](https://raw.githubusercontent.com/BBuf/AI-Infra-Auto-Driven-SKILLS/main/docs/assets/sglang-sota-performance-loop.svg)

`vllm-sota-humanize-loop` 是 vLLM 版本。它先拿 vLLM、SGLang、TensorRT-LLM 做同预算搜索，再判断 vLLM 是否真的落后。如果确实落后，就继续走 profiler、pipeline analysis、必要时的 NCU 证据，最后改 vLLM 源码，避免只在启动参数上来回试。

![vLLM SOTA Humanize Loop](https://files.mdnice.com/user/59/e4302069-eff0-422d-a8f1-5ad856262ec8.png)

这里面有几条规则我觉得很重要：

- 不要拿调过的目标框架去打 competitor 默认参数，每个框架都要 bounded search。
- 模型、精度、GPU 数、workload、SLA 固定之后，再讨论性能差距。
- 先 profile，再用 `llm-pipeline-analysis` 定位到层和 kernel，最后才动源码。
- 如果改的是 CUDA / Triton / CUTLASS 这类 kernel 路径，需要 counter 证据时再接 `ncu-report-skill`。
- 每一轮都要把 benchmark、profile、失败尝试、patch 和复测结果落到 artifact，不然后面很快就会忘掉前面试过什么。

# 0x3. OpenAI Goals 和 4 组 Prompt

OpenAI Codex 的 `/goal` 可以理解成线程级的持久目标。追 SOTA 这种任务跑一轮肯定不够，后面还要 profile、改代码、复测、再判断是否继续。`/goal` 的好处是把完成条件挂在当前线程里：最后要达到什么状态，用什么证据判断成功，哪些约束不能破。官方入口在这里：https://developers.openai.com/codex/cli/slash-commands#set-an-experimental-goal-with-goal

我在 `prompts/` 里放了 4 组 SGLang SOTA prompt：

| Prompt | 用法 |
| --- | --- |
| `sglang-sota-b200-prompts.md` | 普通 skill prompt 版本，面向 B200，覆盖 1/2/4/8 GPU 的模型级 SGLang SOTA 任务。 |
| `sglang-sota-h200-prompts.md` | 普通 skill prompt 版本，面向 H200，适合 `ion8-h200` / `ion9-h200` 这类远端验证环境。 |
| `sglang-sota-b200-codex-goal-prompts.md` | B200 的 Codex `/goal` 版本，把结果、证据、约束、清理规则和停止条件都写进持久目标。 |
| `sglang-sota-h200-codex-goal-prompts.md` | H200 的 Codex `/goal` 版本，适合长时间 benchmark/profile/patch/revalidate 的任务。 |

这些 prompt 里我塞了不少看起来啰嗦、但实际很救命的约束：开始前查相关 open PR；workspace 必须干净；benchmark/profile 前记录 GPU 状态；资源不够就等或者停；只清理当前模型 cache，不动共享 cache；需要提 PR 时只能推到允许的 fork；每个优化 PR 都要写 benchmark 和 GSM8K/MMLU 精度表。

这些细节很烦，但推理框架优化就是这样。少一条，Agent 可能就会拿一组被别人进程干扰的数据当结论，或者把未清理的本地改动混进 patch。

# 0x4. 为什么做这个仓库

我最近用 Codex 做推理框架任务的感受是，很多事情它已经能蹬，但你得给它很具体的上下文和验证方式。尤其是性能优化，不能看到慢就改一行。比较稳的做法一般是先确认 benchmark 是否公平，再拆 prefill/decode，再看 kernel timeline，再查历史 PR，再做一个很小的源码改动，最后回到同一组 workload 复测。这个链条里任何一步偷懒，后面的结论都可能是假的。

`AI-Infra-Auto-Driven-SKILLS` 主要就是把这些流程写下来，交给 Agent 执行，也方便人检查它有没有跑偏。工程判断还是人的事，skill 只是把很多重复、琐碎、容易漏的动作先自动化掉。

如果你在做 SGLang、vLLM、TensorRT-LLM、模型适配、serving 排障或者性能追踪，欢迎试试这个仓库，也欢迎继续贡献新的 AI Infra SKILLS。
